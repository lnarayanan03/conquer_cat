Bug rundown — what's actually breaking

The "bezel coming up at the bottom that changes color with the page/mode" is the iOS safe-area-inset-bottom strip. On iPhones with no home button, iOS reserves ~34 px at the bottom of the viewport for the home indicator. Web pages must paint that area themselves or the browser shows whatever's underneath — which in your app is a flat --app-bg color that doesn't match the gradient/content above it.

P1

Hardcoded black on .main

Line 169 of App.css: .main { background: #000; }. Overrides the per-page bg on desktop & tablet — Calendar's ocean gradient and light theme cream both die.

P1

Safe-area uses wrong variable

--app-safe-fill is defined per-page but nothing actually paints with it. body uses --app-bg, so the home-indicator strip shows the gradient's top color, not the visible bottom color → seam.

P1

100vh on .app

iOS Safari counts 100vh with the URL bar collapsed. With overflow: hidden on .app, content clips past the visible viewport and the last row goes under the home indicator.

P2

iOS auto-zoom on inputs

Your .time-input and .num-input are 16 px. iOS zooms when an input is below 16 px — but you've also got a couple of 14–15 px inputs in the assessment page. Anything below 16 px triggers the zoom.

Patch 4 · iOS input zoom P2

iOS auto-zooms when an input's font-size is < 16 px on focus. Make every input explicitly 16 px so this never happens.

src/App.css · global rule 
@@ append near the top, after :root blocks @@
+/* Never let iOS auto-zoom on focus */
+input, textarea, select {
+  font-size: 16px;
+}
+@media (min-width: 768px) {
+  input, textarea, select { font-size: inherit; }
+}

Patch 5 · responsive breakpoints P2

You only have two breakpoints: @media (max-width: 1023px) and @media (max-width: 560px). Tablets (iPad portrait = 820 px) fall into the "mobile" bucket but have plenty of room. Add a middle breakpoint and let the sidebar return on iPad landscape.

src/App.css · add new breakpoints 
@@ add after @media (max-width: 1023px) block @@
+/* iPad landscape and small desktops — bring the sidebar back */

+  .sidebar { display: flex; width: 180px; min-width: 180px; }
+  .bottom-nav, .mobile-header { display: none !important; }
+  .main { padding-top: 0 !important; padding-bottom: 0 !important; }

+/* Tablet portrait — wider gutters, larger cards */

Patch 9 · Rubber-band scroll exposing body bg P2

When the user overscrolls the .main column on iOS, the page rubber-bands and shows the body background underneath the content. Combined with the bezel bug, this looked like a brief "color flash" at the top during pull-to-refresh-style gestures. overscroll-behavior stops the chain so the rubber-band stays inside the scroller.

Patch 10 · Stop imperative --app-bg setter P3

CalendarPage (App.jsx line ~3097) runs a useEffect that sets document.documentElement.style.backgroundColor and overrides --app-bg and --app-safe-fill on every voyage-mode / theme change. It also attaches a MutationObserver on document.body to watch data-theme. This works, but is fragile:

  • It bypasses the CSS cascade — once any rule loses, JS has to re-patch.
  • It races other components that read --app-bg on mount.
  • The mutation observer fires on every body-attribute change, not just data-theme.
  • Leaving Calendar doesn't restore the variables — they linger as inline styles on html.

Replace it with a single body class and pure-CSS rules — which the App.css already has at lines 19–49.

How to apply

  1. Pull a fresh branch: git checkout -b fix/mobile-safe-area
  2. Apply patches 1–3 first (the bezel fixes) — these alone resolve what you're seeing.
  3. Run npm run dev, open on an actual iPhone (or iOS Simulator), confirm the strip is gone on every page in both dark/light + calendar mode.
  4. Add patches 4–6 once you can ship the urgent fix.
  5. Tip for testing: in iOS Simulator,  Device → Erase All Content and Settings  doesn't update the home-screen icon's theme-color; you have to remove and re-add the PWA.
